可以把 **Gateway** 理解成 OpenClaw 的“总调度中心”。

你在飞书里发一句话，看起来只是“消息进来，AI回一句”，但 Gateway 在中间其实做了很多事：**接消息、判断交给谁、排队、找上下文、驱动 Agent 干活、再把结果发回去**。  
下面我按“先整体，再走一遍飞书消息链路”的方式，给你重新梳理。


---

# 二、用最通俗的话理解：飞书发来一条消息后，到底发生了什么

假设你在飞书里发了一句：

> 帮我查一下今天的股票行情

从你点击发送，到 OpenClaw 回复再返回到你的飞书，中间大致要经过 **6 步**。

---

## 第 1 步：飞书把消息交给 Gateway

你在飞书里发了个消息，消息会先发送到飞书的服务器。  
然后飞书通过插件，把这条消息推送给你部署的OpenClaw的Gateway。

这时 Gateway 拿到的，不只是文字本身，还会拿到一堆“附带信息”，比如：

- 这是哪个渠道来的：飞书
    
- 是谁发的
    
- 是私聊还是群聊
    
- 群 ID / 用户 ID 是多少
    
- 目标 bot 账号是谁
    
- 消息内容是什么
    

你可以把这一步理解成：

**Gateway 先当“前台接待员”，把外部世界送来的消息接收进来，并拆成结构化信息。**

---

## 第 2 步：Gateway 判断这条消息该交给哪个 Agent

消息到了以后，不是马上就给 AI。  
Gateway 会先判断：

> 这条消息应该由谁来处理？

比如你可能配置过：

- 飞书私聊 → main Agent
    
- 某个写作群 → writer Agent
    
- 某个Code群 → code Agent
    

这一步是**规则匹配**，不是让模型去猜。

也就是说，Gateway 会根据配置看：

- 来自哪个渠道
    
- 哪个账号
    
- 私聊还是群聊
    
- 是哪个具体群
    
- 是否线程继承上一个上下文
    

然后用最精确的规则去匹配目标 Agent。

### 为什么这样设计很聪明

因为它有三个优点：

#### 1. 稳定

不会今天 AI 觉得该给 main，明天又觉得该给 writer。

#### 2. 快

不用先额外跑一次模型来做“路由判断”。

#### 3. 省钱

不消耗额外 token。

所以这一步你可以理解成：

**Gateway 像分诊台，先把病人分到正确科室，而不是让 AI 自己猜该挂哪个号。**

---

## 第 3 步：消息先排队，而不是立刻执行

确定交给哪个 Agent 后，Gateway 通常也不会马上开始处理，而是先进入队列。

为什么要排队？

因为同一个 Session 里，通常只能有一个 Agent 正在运行。  
否则会乱。

比如你连续发三句：

1. 帮我查今天日志
    
2. 顺便看下数据库状态
    
3. 如果有异常发我摘要
    

如果三条同时并行跑，可能会出现：

- 第二条没等第一条结果
    
- 第三条不知道前两条执行到了哪
    
- 工具操作互相打架
    

所以 Gateway 采取的思路是：

- **不同会话可以并行**
    
- **同一个会话内部严格串行**
    

这就像餐厅后厨：

- 不同桌的单子可以同时做
    
- 但同一桌加单，得按顺序处理，避免前后逻辑错乱
    

### 队列还有几种模式

文章里提到 collect、steer、followup 等模式。通俗说：

- **collect**：把短时间内来的多条消息合并一下再处理
    
- **followup**：老老实实排在后面等
    
- **steer**：当前 AI 正在干活时，你插一句补充要求，它能在合适时机“插入”当前运行流程
    

其中 steer 很像：

> AI 正在埋头做事，你从旁边补一句：“对了，优先查 error 日志。”

它不是完全打断，而是在下一个合适节点把你的补充塞进去。

---

## 第 4 步：Gateway 找到这条消息属于哪个 Session

这里是最容易被忽略，但实际非常关键的一步。

**Session = 这次对话所属的上下文容器。**

Agent 回答问题，不是只看“这一句”，而是看这句话属于哪段对话、接着谁的上下文说。

Gateway 会判断：

- 这是私聊吗？
    
- 是某个群吗？
    
- 是哪个用户？
    
- 是否要跨渠道复用上下文？
    
- 是否该新开一个 Session？
    
- 这个旧 Session 是否已经过期？
    

### 举几个例子

#### 情况 1：你自己一个人用

可能飞书私聊、Telegram 私聊、WebChat 私聊，全都汇入同一个主 Session。  
这样你在飞书说了一半，去 Telegram 接着聊，也能连续上。

#### 情况 2：多人共用一个 bot

那通常会按“每个用户单独隔离”来建 Session。  
否则 A 说的话会混到 B 的上下文里。

#### 情况 3：群聊

群聊天然按群隔离。  
这个群一个 Session，另一个群另一个 Session。

### Session 还会重置

比如：

- 每天凌晨 4 点后开新上下文
    
- 空闲 30 分钟后重置
    
- 你手动发 `/new` 或 `/reset`
    

注意，这种“重置”不是删记录，而是：

- 旧的归档
    
- 新的开始
    

就像翻到新的一页，但旧笔记还保留着。

所以这一步可以理解成：

**Gateway 在给消息找“它应该接在哪段对话后面”。**

---

## 第 5 步：真正进入 Agent Loop，AI 开始干活

到了这里，消息才真正交给 Agent。

这时 Gateway 会做一件非常关键的事：**组装上下文**。

AI 并不是拿着一句“帮我查日志有没有异常”就直接开始回答。  
Gateway 会把很多东西一起塞进去，构成完整认知环境。

通常包括：

- **SOUL.md**：这个 Agent 是谁，性格、边界、说话风格
    
- **USER.md**：你是谁、你的习惯和偏好
    
- **AGENTS.md**：这个 Agent 有什么能力
    
- **MEMORY.md**：长期记忆（一般只在私聊场景加载）
    
- **今天/昨天的日记**
    
- **Skills 快照**
    
- **历史对话**
    
- **这条新消息**
    

然后进入标准的 Agent Loop：

### Agent Loop 通俗版

1. 把上下文全部准备好
    
2. 把新消息放进对话历史
    
3. 调用大模型开始思考
    
4. 如果模型觉得要用工具，就去调用工具
    
5. 工具返回结果后，再交给模型继续想
    
6. 重复这个过程，直到模型产出最终回答
    
7. 把回答流式发出去
    

比如你说“查日志有没有异常”，模型可能会决定：

- 先调用 read / grep / exec 之类工具
    
- 读取日志
    
- 分析异常模式
    
- 整理成摘要
    
- 最后回复你
    

### 为什么你会看到“AI 正在输入”

因为 Gateway 用的是“两阶段响应”思路：

#### 第一阶段：立刻确认收到

Gateway 先告诉客户端：

> 收到了，已经开始处理

所以飞书可以先显示“正在输入”。

#### 第二阶段：边处理边推送结果

Agent 干活的过程中，工具调用、文本生成、最终结果，会通过 WebSocket 流式推送出去。

所以你看到的不是“等很久突然冒出整段话”，而是：

- 先显示在处理中
    
- 然后回复一点点出现
    
- 最后完整结束
    

这一步你可以理解成：

**Gateway 像项目经理，把资料准备好，叫 Agent 开工，必要时调工具，再把产出一点点往外送。**

---

## 第 6 步：把 AI 回复发回飞书

当 Agent 生成回复以后，Gateway 还不能直接原样扔回去。  
它还要经过“渠道适配”。

因为不同渠道的消息格式不一样：

- 飞书有飞书的消息格式
    
- Telegram 有 Telegram 的格式
    
- Discord 有 Discord 的格式
    

所以 Gateway 会把 Agent 生成的内容，交给飞书对应的渠道插件做转换，然后再通过飞书连接发回去。

最终你在飞书里看到的，就是处理后的回复。

如果是长回复，还可能边生成边分块发送。  
所以你看到的“逐渐出现”，本质上就是：

**模型在生成，Gateway 在转发，飞书在展示。**

# 三、把整个过程压缩成一条最容易懂的链路

你可以直接记这一版：

### 飞书发消息 → Gateway 收到 → 判断交给哪个 Agent → 排队 → 找到对应 Session → 组装上下文 → Agent 调模型和工具干活 → 结果流式返回 → Gateway 转成飞书格式 → 发回飞书

这就是最核心的一条链。

# 四、再用一个生活化比喻讲一遍

把 Gateway 想成一家餐厅的“总控台”，会更好懂。

### 你在飞书发消息

相当于顾客进店点单。

### Channel 接收

前台先把单子收下来，识别你是谁、在哪桌、点了什么。

### 路由

前台判断这单该给哪个厨师：

- 家常菜给 main
    
- 写作类给 writer
    
- 运维类给 ops
    

### 队列

后厨不是每张单都立刻做，要按桌次和优先级排队，避免同一桌菜乱套。

### Session

总控台确认这张单属于哪一桌、接的是哪一轮菜。

### Agent Loop

厨师开始做菜：

- 看菜单
    
- 看你之前点过什么
    
- 看这桌口味偏好
    
- 需要时去仓库拿材料（调用工具）
    
- 最后把菜做好
    

### 回复发送

服务员把菜端回你桌上，也就是把回复发回飞书。

所以 Gateway 本质上就是：

**前台 + 调度台 + 后厨总控 + 配送中枢**

# 五、WebSocket 在这里是干什么的

文章里还讲了 WebSocket 协议，这部分你可以这样理解：

Gateway 和外部客户端（比如 WebChat、CLI、macOS App、Node 设备）之间，不是每次都重新发 HTTP 请求，而是保持一条**长连接**。

这条长连接有几个作用：

### 1. 建立连接后马上做身份确认

谁连进来了？  
你是 operator 还是 node？  
有没有权限？  
是不是已配对设备？

### 2. 可以双向实时通信

客户端可以发请求给 Gateway，Gateway 也可以主动推事件。

### 3. 适合流式结果

AI 回复不是一次性返回，而是边生成边推。  
WebSocket 天然适合这种实时交互。

### 4. 适合设备在线管理

比如手机、电脑、树莓派这些 Node 设备连上来后，可以持续声明：

- 我有摄像头
    
- 我能录屏
    
- 我能定位
    
- 我能执行某些命令
    

Gateway 就能统一调度它们。

所以 WebSocket 在 Gateway 里的角色，可以理解成：

**一根一直不断开的实时通信管道。**

# 六、为什么说 Gateway 不是黑盒

理解完上面这条链路之后，你排查问题就有方向了。

### 1. 收不到消息

先看 Channel 层。  
可能是飞书插件没接好，或者 WebSocket 断了。

### 2. 消息进来了，但跑错 Agent

看路由配置 binding。  
多半是匹配规则没写对。

### 3. 回复特别慢

可能有两种原因：

- 队列里前面还有消息在排
    
- Agent 推理 / 工具执行本身比较慢
    

### 4. 上下文接不上

看 Session 定位和重置策略。  
可能 Session 被重置了，或者隔离范围配错了。

### 5. AI 回了，但飞书没显示

看回复发送链路。  
可能是渠道插件格式转换或发送出了问题。

也就是说，Gateway 一旦拆开看，它其实不是神秘黑盒，而是一条很清楚的流水线。

# 七、给你一个更适合讲给别人听的“精简版”

你要是想用很通俗的话对别人解释，可以直接这么说：

> OpenClaw 里的 Gateway，就像整个系统的总调度中心。  
> 飞书消息一来，先由 Gateway 接住；然后它根据规则判断该交给哪个 Agent；接着把消息排进队列，找到对应的对话上下文 Session；再把人格、记忆、历史对话、工具能力等上下文组装好，驱动 Agent 去调用模型和工具处理；最后把生成结果通过飞书插件转成飞书能显示的格式，再发回飞书。  
> 整个过程中，Gateway 还负责消息路由、会话隔离、并发控制、流式回复、设备管理和定时任务，所以它其实是 OpenClaw 的核心中枢，而不只是“跑 Agent 的地方”。
