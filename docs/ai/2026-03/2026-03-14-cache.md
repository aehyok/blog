Cache read 和 cache write 是 Anthropic **Prompt Caching** 功能的两个概念，用来节省 token 费用和提升响应速度。

## 基本原理

当你用 Claude Code 处理大型代码库时，每次请求都需要把上下文（代码文件、系统提示等）发送给 API。Prompt Caching 允许把这些内容"缓存"起来，避免重复处理。

## Cache Write（缓存写入）

- 第一次把某段内容发送给 API 时，Anthropic 会将其存入缓存
- 这次请求会产生 **cache write token** 的费用
- 费用比普通 input token **略贵一点**（大约 1.25x），因为需要存储开销
- 缓存有效期通常是 **5 分钟**（TTL 会在每次命中时刷新）

## Cache Read（缓存读取）

- 后续请求中，如果包含相同的内容，API 直接从缓存读取，不重新处理
- 产生 **cache read token** 的费用
- 费用比普通 input token **便宜很多**（大约 0.1x，即九折优惠）
- 响应速度也更快

## 在 Claude Code 中的实际意义

```
第一次分析项目：
  → 读取大量代码文件 → Cache Write（稍贵）

后续对话/修改：
  → 相同代码上下文直接复用 → Cache Read（很便宜）
```

Claude Code 在处理大型代码库时会自动利用这个机制，所以你会看到账单或日志里区分这两种 token 类型。

## 为什么要区分？

计费不同，所以 Anthropic 在用量统计里分开显示，让你清楚地看到实际节省了多少费用。如果一个会话里 cache read 占比很高，说明缓存命中率好，成本效率高。

**缓存是临时的**，默认 TTL 是 **5 分钟**，每次命中会刷新计时。超时后缓存自动失效。

**新开会话后**，缓存不会延续。第一次发送上下文又会产生 cache write，后续同一会话内的对话才会 cache read。

所以 Claude Code 的缓存收益主要体现在**同一个长会话内**的多轮对话，而不是跨会话复用。
