# 为什么输出 Token 比输入 Token 贵？
来源：大饶不老
作者：大饶
发布时间：2026/03/25 21:56:18
原文链接：https://mp.weixin.qq.com/s/8cUmlE4AcD4woGmdh-HHgQ
![cover](./output-token-cover.jpg)
> 在使用 GPT-4、Claude 等大模型 API 时，你可能会发现：输出 Token 的价格往往是输入 Token 的 2-5 倍。这背后到底有什么技术原因？本文将深入剖析。

## 一、直观对比：价格差异有多大？

以主流大模型 API （非最新版本）的定价为例：

| 模型 | 输入 ($/1M tokens) | 输出 ($/1M tokens) | 输出/输入倍数 |
| --- | --- | --- | --- |
| GPT-4 | $30 | $60 | 2x |
| Claude 3 Opus | $15 | $75 | 5x |
| GPT-4o | $2.50 | $10 | 4x |

**结论**：输出 Token 的定价普遍是输入 Token 的 2-5 倍。GPT-4o输入相比GPT-4已大幅下降，但4o是推理增强模型（推理要进行多次思考输出），输出相比输入价格倍数增加了。

## 二、核心原因：计算模式的本质差异

### 1. 输入阶段：一次性并行处理

```
用户输入: "请解释量子力学"
↓
[一次性编码]
↓
并行处理所有 Token
↓
计算量: O(n²) 或 O(n)
```

**特点**：

- 所有输入 Token 同时进入模型

- 可以高度并行化计算

- 只需一次完整的前向传播

### 2. 输出阶段：串行自回归生成

```
模型输出: "量子" → "力学" → "是" → "研究" → "微观" → ... → [结束]
↓ ↓ ↓ ↓ ↓
第1步 第2步 第3步 第4步 第5步

每步都需要完整的前向传播！
计算量: O(m × n²)，其中 m = 输出长度
```

**特点**：

- 必须逐个 Token 生成

- 每个新 Token 依赖之前所有已生成内容

- 无法并行，只能串行执行

## 三、技术原理解析

### 自回归生成的计算开销

大语言模型采用**自回归（Autoregressive）**生成方式：

```
P(x_t | x_1, x_2, ..., x_{t-1})

含义：生成第 t 个 Token 时，必须知道前面 t-1 个 Token
```

这导致：

- 生成长度为 100 的回复，需要执行 **100 次**完整的前向传播

- 每次都要重新计算整个上下文的注意力机制

- 虽然 KV Cache 能优化部分计算，但无法消除串行依赖

### 注意力机制的复杂度

```
标准自注意力计算复杂度: O(n² × d)
- n = 序列长度
- d = 隐藏维度

输入阶段：计算一次 Q·K^T。
输出阶段：每生成一个 Token 都要更新并重新计算注意力，
```

### 内存带宽瓶颈：

```
┌────────────────────────┐
│ 模型权重（静态，可复用）
│ 输入时加载一次
├────────────────────────┤
│ KV Cache（动态增长）
│ 每生成一个 Token 都要频繁读写
│ 内存带宽成为性能瓶颈
└────────────────────────┘
```

**关键问题**：

- 现代 GPU 的算力增长快于内存带宽增长

- 输出阶段往往是**内存带宽受限**而非算力受限

- 频繁读写 KV Cache 导致效率降低

（这就是PD分离背后的原因，输入是计算型，输出是频繁访存型，老两者对资源的要求不同）

## 四、为什么无法完全并行化？

### 1. 自回归的数学本质

```
原因：条件概率的链式依赖
P(x_1, x_2, ..., x_m) = P(x_1) × P(x_2|x_1) × P(x_3|x_1,x_2) × ...

每个 Token 的生成必须等待前一个完成
```

### 2. 采样策略的多样性

| 策略 | 说明 | 并行难度 |
| --- | --- | --- |
| Greedy | 每步取概率最高的 Token | 低 |
| Sampling | 按概率分布随机采样 | 中 |
| Beam Search | 维护多个候选序列 | 高 |

不同的采样策略增加了并行化的复杂度。

### 3. 动态停止条件

- 模型不知道何时应该生成结束标记 `[EOS]`
- 无法预先分配计算资源
- 必须在运行时动态决定生成长度

## 五、时间 = 金钱

### 用户体验的实时性要求

| 阶段 | 延迟敏感度 | 用户感知 |
| --- | --- | --- |
| 输入处理 | 低 | 一次性等待 |
| 输出生成 | 高 | 持续感知 |

- 输入阶段：用户发送后等待，可以接受几百毫秒
- 输出阶段：用户看着文字**逐字出现**，延迟必须控制在几十毫秒内

### 批处理效率差异

```
输入批处理：
┌─────┬─────┬─────┬─────┐
│ Q1 │ Q2 │ Q3 │ Q4 │ → 一次计算，效率极高
└─────┴─────┴─────┴─────┘

输出批处理：
┌─────┐ ┌─────┐ ┌─────┐
│ A1 │ → │ A2 │ → │ A3 │ → 难以并行，效率低
└─────┘ └─────┘ └─────┘
```

**结论**：相同硬件下，输出吞吐远低于输入吞吐。

## 六、总结

| 因素 | 输入 Token | 输出 Token |
| --- | --- | --- |
| 计算次数 | 1 次 | N 次（N=输出长度） |
| 并行度 | 高 | 低（串行依赖） |
| 内存访问 | 少 | 频繁（KV Cache） |
| 延迟敏感度 | 低 | 高 |
| 批处理效率 | 高 | 低 |
| 实际计算成本 | 基准 | 3-10 倍 |
| API 定价倍数 | 1x | 2-5x |

## 七、总结

> **输入是"一次性阅读"，输出是"逐字手写"。**
> 后者需要反复思考、动笔、检查，自然更耗时、更耗资源、更贵。

大模型 API 对输出 Token 收取 2-5 倍于输入 Token 的费用，实际上已经是一个**优惠后的价格** —— 真实的计算成本差异可能高达 3-10 倍。
