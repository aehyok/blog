这是claude官网博客给MCP和Skills的一个解释：MCP connects Claude to data，Skills teach Claude what to do with that data。

用中文通俗的来说就是：MCP 负责让大模型（Claude） 拿到数据，Skills 则负责告诉大模型（ Claude）怎么用这些数据。

>宝玉文章中有一句话总结的很精辟：Skills 描述工作流程，MCP 提供执行引擎。但很多时候，操作系统自带的引擎就够用了。

接下来我先来看看MCP和Skills各自的定义。
## MCP的定义和功能
- MCP的定义： MCP（Model Context Protocol）是一种开放协议，用于标准化大模型与外部工具、数据源、系统之间的连接方式。

![[Pasted image 20260124110610.png]]

实现一个MCP包括哪几部分呢



MCP在使用的时候要注意，要注意上下文的占用问题，我一般是要使用的MCP就去开启它，而不是让MCP开启，因为MCP的存在就会占用我们上下文的token。

![[Pasted image 20260124160025.png]]

拿宝玉给的数据：
- GitHub MCP Server：27 个工具，消耗约 18,000 tokens
- Playwright MCP Server：21 个工具，消耗约 13,600 tokens
- mcp-omnisearch：20 个工具，消耗约 14,200 tokens
如果再多几个工具，先不说Token花费的问题，主要是上下文很快就被占满，然后就开始胡说八道了。

当然在Claude Code中推出了Tool Search的功能，用于按需发现MCP的插件，也就是AI需要某个MCP工具的时候，先搜索再进行加载，极大的减少了预加载MCP工具描述到上下文中

接下来Skills登场，能否解决很大一部分MCP的问题
## Skills的定义和功能
- Skills的定义：类似于专业的培训手册。它们是包含指令、脚本和资源文件的文件夹，旨在教会大模型如何执行特定的任务。大模型就会根据任务需求动态的加载这些技能。

![[Pasted image 20260124111248.png]]

来详细说一下一个Agent Skill具体包含哪些东西

![[Pasted image 20260124163330.png]]

第一层：元数据（启动时加载）

- 我标记的1 
- 只有名称和简短描述
- 每个 Skill 约 100 tokens
- 装 100 个 Skill 也只占 10,000 tokens
    
第二层：完整指令（相关时加载）

- 当 AI 判断某个 Skill 与任务相关时，才读取完整的[SKILL.md](https://skill.md/)
- 建议控制在 5,000 tokens 以内

第三层：参考资料references（需要时加载）
- 详细的技术文档、API 说明、示例代码
- 按需读取，用多少加载多少
- 理论上可以包含无限内容
- 也就是SKILL.md中要对references中的引用要进行说明

> 这里着重强调一下scripts，它就是脚本，可以是shell代码、js代码、python代码、go代码等等，也就是它是一个可以执行的脚本就行了。
> 脚本本身只参与到执行，它本身是不会被加载到上下文中的，这里可以放心，所以理论上这个脚本是可以很复杂的。

回顾一下开头宝玉文章中那句精辟的话语：Skills 描述工作流程，MCP 提供执行引擎。但很多时候，操作系统自带的引擎就够用了。

> 我的个人理解就是：比如window系统本身他就有cmd命令，读写文件的基础工具，对于很多本地的任务，Skill下的Script写一些轻量的脚本就可以实现很多本地化的操作任务了，在原来没有Skills的时候，MCP的作用就很突然。但是现在很多MCP的场景都可以使用Skill+脚本就可以轻松实现。

那么现在script都可以适配哪些场景呢

- 读写本地文件 → bash + Skill 脚本
- 处理 PDF/Word/Excel → Skill 脚本
- 运行代码分析 → Skill 脚本
- 执行 Git 操作 → Skill 脚本
- 生成图表和可视化 → Skill 脚本
- 优化自己或团队的工作流

仍然需要MCP的场景又有哪些呢

- 连接远程各种系统获取各种数据
- 调用第三方 SaaS API（Slack、Notion、Jira）
- 查询云端数据库
- 访问需要认证的外部服务
- 做一个服务让外部用户都能用

在使用Skills的过程中，我们也会发现，当我们在原有新的Skill中有了新的需求或者是遇到了bug，那么这个skill它就可以进入到自进化的过程，进行自我的升级改造，或者修复其中存在bug。这一点也算是跟MCP的一个本质的区别。而远程的云端MCP服务，相当于小黑盒，我们是完全无法修改的，而本地的skill，或者别人的skill我们安装获取之后，就可以在本地进行加工处理，达到符合我们自己的需求。(当然原本你只是在本地写的mcp服务工具，大部分情况下直接用skill是完全可以实现的，当然可能有特例，目前我还没发现)

最后我们再来看看MCP和Skills的相同点和不同点

## MCP和Skills的相同点

![[Pasted image 20260124180227.png]]

- 增强大模型的能力：两者都能够扩展大模型能力，使其超越了简单的文本对话，能够处理更复杂、更实际的工作流。
- 都可以包含代码的技术实现：Skills技能下可以包含可执行的代码脚本或参考文件，同样MCP服务的实现也是通过代码技术进行实现的。
- 协同工作：Skills中可以调用MCP服务，但MCP服务中是不能调用Skills的。可以进行组合使用，但不能随意乱用。
- 复用性：两者都是通过标准化的方式来解决重复性的问题，而不是每次都在提示词中重新定义。

我这里只列举了我觉得比较重要的。

## 最后

不知道你是否搞懂了MCP和Skills这两个概念。真的自己写一遍之后，确实懂了一些皮毛了。如果再遇到不懂的知识就照着宝玉的博客把它写一遍。

之前做过两个前端开发使用的MCP，一个是随时添加菜单信息，以及给角色添加菜单权限，另外一个是根据后端提供的Swagger来添加前端的接口定义和结构，因为涉及到多个后端微服务，开源的MCP确实是无法满足。

感觉上swagger这个完全可以替换成脚本，直接进行适配。
但是随时添加菜单以及进行授权的，它是完全可以开放给其他同事，或者如果项目足够大，或者是个平台，也完全可以开放给外部进行使用，这也是MCP的优势所在。

最后如果有了重复性 的工作，可以优先考虑使用skill进行解决，如果发现他好像好像skill满足 不了，那么就可以考虑mcp了。


| 维度       | Skills（技能）       | MCP（模型上下文协议）           |
| -------- | ---------------- | ---------------------- |
| 核心关注     | **如何做**          | **能做什么**               |
| 解决问题     | 事情**怎么做得对、做得好**  | 事情**能不能做、能做到哪一步**      |
| 本质       | 程序性知识（流程、方法、规范）  | 能力与资源暴露（工具、数据、系统 API）  |
| 对模型的影响   | 改变**思考方式与决策流程**  | 扩展**可执行能力与感知范围**       |
| 是否直接产生行动 | ❌ 不直接产生真实行动      | ✅ 可以直接读 / 写 / 操作外部系统   |
| 是否依赖外部系统 | 否（可纯 Prompt 存在）  | 是（必须连接真实世界）            |
| 典型内容     | SOP、规则、示例、判断标准   | API、数据库、文件系统、SaaS 工具   |
| 典型用途     | 代码审查规范、报告格式、分析流程 | 查文件、查库、调接口、写数据         |
| 加载方式     | 按需加载（On-demand）  | 持续可用（Always available） |
| 粒度       | 粗粒度（角色 / 任务级）    | 细粒度（能力 / 操作级）          |
| 复用方式     | 跨任务复用“做事方法”      | 跨 Agent 复用“工具能力”       |
| 单独使用的效果  | 会想、会说，但可能做不了     | 能做，但容易乱做               |
| 最佳协作关系   | 决定**该怎么做**       | 提供**能做哪些事**            |
